## 1. Study Registration Moves To Go

Phase 10 replaces the phase 9 placeholder for `open_study` with a real Go-backed registration flow.

The Go backend now owns:

- backend-generated study IDs
- in-memory study registry state
- `inputPath` echoing in the returned payload
- `inputName` extraction from the opened path
- contract-shaped `OpenStudyCommandResult` responses

This keeps the frontend runtime contract stable while moving the first stateful backend responsibility out of Rust.

## 2. Transport Behavior

`POST /api/v1/commands/open_study` now:

- requires a JSON body matching the frozen `OpenStudyCommand`
- rejects unknown fields and malformed payloads as `invalidInput`
- validates that `inputPath` exists and points to a file
- registers the study in the Go registry
- returns `200 OK` with `{ study: ... }`

The route now returns backend-style errors with contract v1 error codes instead of the old generic `501 Not Implemented` placeholder.

## 3. Recent-Study Recording Hook

The Go persistence package now includes a real recent-study catalog writer instead of phase 8/9 directory scaffolding only.

On successful `open_study`:

- the catalog deduplicates by `inputPath`
- the newest entry is inserted first
- the list is truncated to 10 entries
- corrupt JSON is renamed aside as `catalog.corrupt.json`

To preserve current Rust behavior, persistence write failures are logged but do not cause `open_study` itself to fail.

## 4. Metadata Boundary Kept Narrow

Phase 10 deliberately stops short of a general Go DICOM metadata reader.

The returned `StudyRecord` already preserves the contract field for `measurementScale`, but phase 10 does not populate it yet. That keeps the study-registry transfer isolated from the risk called out in the migration plan: letting measurement-scale extraction drag full DICOM metadata work forward too early.

This is consistent with the repository’s included sample parity fixture:

- [backend/tests/fixtures/parity/sample-dental-radiograph/describe-study.json](backend/tests/fixtures/parity/sample-dental-radiograph/describe-study.json)

That fixture currently contains no measurement-scale payload, so phase 10 can move study ownership to Go without claiming that phase 11’s metadata-reader work is already complete.

## 5. Validation

Validate phase 10 with:

```bash
env GOCACHE=/tmp/xrayview-go-build-cache GOPATH=/tmp/xrayview-go-path go -C go-backend test ./...
curl -s -X POST http://127.0.0.1:38181/api/v1/commands/open_study \
  -H 'content-type: application/json' \
  -d '{"inputPath":"'"$(pwd)"'/images/sample-dental-radiograph.dcm"}'
```

Expected `open_study` behavior:

- `study.studyId` is generated by Go
- `study.inputPath` matches the requested path
- `study.inputName` is `sample-dental-radiograph.dcm`
- the Go runtime `studyCount` increments after registration

## 6. Exit Criteria Check

Phase 10 exit criteria are now met:

- `openStudy` can run through the Go backend transport
- study IDs are generated in Go
- Go owns the study registry and response model
- input path validation and input name extraction happen in Go
- the recent-study recording hook exists on the Go side

Phase 11 remains the place to prove Go-side DICOM metadata extraction instead of smuggling that larger subsystem into this registry phase.
